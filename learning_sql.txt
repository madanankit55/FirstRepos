5) basic architecture of PySpark

The basic architecture of PySpark consists of three main components:
Driver:
The driver is the main component of PySpark. It is responsible for creating the SparkContext, which is the entry point to Spark. The driver also manages the execution of Spark applications.
Executors:
Executors are responsible for running the tasks that are assigned to them by the driver. Executors are typically located on worker nodes in the Spark cluster.
Cluster Manager:
The cluster manager is responsible for managing the cluster of worker nodes. It allocates resources to worker nodes and monitors their health.
PySpark applications are written in Python and compiled to run on the JVM. Spark applications can be run in various modes, including local mode, cluster mode, and client mode.
Local mode is the simplest mode to run PySpark applications. In local mode, the driver and executors run on the same machine. This mode is useful for testing and debugging PySpark applications.
Cluster mode is the most common mode to run PySpark applications. In cluster mode, the driver runs on the client machine and the executors run on worker nodes in the cluster. This mode is used to run PySpark applications on large datasets.
Client mode is similar to cluster mode, except that the driver remains on the client machine that submitted the application. This mode is useful for running PySpark applications on a cluster that is managed by a third party.
PySpark provides a variety of APIs for data processing, including RDDs, DataFrames, and Datasets. RDDs are the basic data structure in PySpark. They are immutable distributed collections of data. DataFrames are a higher-level abstraction that provides a more structured view of data. Datasets are a more recent abstraction that provides a type-safe and optimized view of data.
PySpark also provides a variety of libraries for machine learning, graph processing, and streaming data processing. These libraries make it easy to build and deploy powerful data processing applications.

The PySpark architecture is designed to be scalable and fault-tolerant. PySpark applications can be run on clusters of thousands of nodes. PySpark also provides a variety of features that make it easy to recover from failures.
PySpark is a powerful tool for processing large datasets in a distributed manner. It provides a high-level interface that makes it easy to write and run Spark applications. PySpark is also scalable and fault-tolerant, making it a good choice for production applications.

ii) Salting : It involves adding a random component, or ‘salt’, to the data to distribute it more evenly across partitions. 
This results in a more balanced workload and improved performance.

from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.types import StringType

# Create a Spark session
spark = SparkSession.builder.appName("skewed_data_example").getOrCreate()
# Create a list with skewed data
skewed_data = ['A'] * 1000 + ['B'] * 100 + ['C'] * 10 + ['D']
# Create a DataFrame with skewed data
data = [(key,) for key in skewed_data]
df = spark.createDataFrame(data, ["key"])

# Show the initial distribution of the 'key' column
print("Initial Distribution:")
df.groupBy("key").count().show()

# Identify the skewed key
skewed_key = df.groupBy('key').count().orderBy('count', ascending=False).first()[0]

# Add a random salt to the skewed key
df = df.withColumn('salted_key', F.when(F.col('key') == skewed_key, F.concat(F.col('key'), F.lit('_'), F.rand())).otherwise(F.col('key')))

# Repartition the data based on the salted key
df = df.repartition('salted_key')

# Show the final distribution after salting
print("Distribution After Salting:")
df.groupBy("salted_key").count().show()

# Stop the Spark session
spark.stop()

iv) Bucketing: Bucketing works by specifying a column and a number of buckets during the creation of the DataFrame. Spark then applies a hash function to the specified column 
and divides the data into buckets corresponding to the hash values. The number of buckets remains fixed, so the distribution of data doesn’t change with the size of the data

numBuckets = 100
salesData.write.bucketBy(numBuckets, "customerId").sortBy("transactionId").saveAsTable("salesData_bucketed")
Now, when you run a JOIN operation between salesData and another DataFrame customerInfo:

customerInfo = spark.table("customerInfo")
bucketedSalesData = spark.table("salesData_bucketed")
result = bucketedSalesData.join(customerInfo, bucketedSalesData.customerId == customerInfo.customerId)

Because salesData is bucketed by customerId, Spark can execute the join more efficiently. It knows that all rows with a particular customerId are located in the same bucket,
so it can avoid shuffling all the data across the network and perform the operation on each bucket.
This can drastically reduce the time and resources needed for this operation, especially for large datasets.

OLAP (Online Analytical Processing) vs OLTP (Online Transaction Processing)

1) OLAP is online database query management system , where OLTP is online database modification system 
2) OLAP consist of historical data from various database, where OLTP consists of transaction data.
3) OLAP is used for data mining, analytics and decision making where OLTP used for business task.
4) OLAP serves the purpose of extract information for analysis and decision making where OLTP serves the purpose of CRUD operations.
5) OLAP needs backup time to time as compared with OLTP, where OLTP takes backup and recovery process is maintained very rigorously.
6) OLAP is mostly used for reading the data , its rarely the data, where OLTP is does CRUD operations.

select d.Name as Department, a. Name as Employee, a. Salary 
from (
select e.*, dense_rank() over (partition by DepartmentId order by Salary desc) as DeptPayRank 
from staging.Employee e 
) a 
join staging.Department d
on a.departmentId = d.Id 
where DeptPayRank <=3;

--- aggregating the sum of all values
select salesperson_id,order_number,order_date,cust_id,sum(amount) over () from staging.int_orders
-- aggregating the sum based on partitioning condition 
select salesperson_id,order_number,order_date,cust_id,sum(amount) over (partition by salesperson_id) from staging.int_orders
--doing all sum previous value to next value
select salesperson_id,order_number,order_date,cust_id,amount,sum(amount) over (order by order_date ) from staging.int_orders
--  doing all sum based partition condition
select salesperson_id,order_number,order_date,cust_id,amount, sum(amount) over (partition by salesperson_id order by order_date) from staging.int_orders  
-- previous 2 rows sum like that example
select salesperson_id,order_number,order_date,cust_id,amount,sum(amount) over (order by order_date rows between 2 preceding and current row ) from staging.int_orders
-- previous 2 and 1 preceding
select salesperson_id,order_number,order_date,cust_id,amount,sum(amount) over (order by order_date rows between 2 preceding and 1 preceding) from staging.int_orders
-- 1 preceding and 1 following
select salesperson_id,order_number,order_date,cust_id,amount,sum(amount) over (order by order_date rows between 1 preceding and 1 following ) from staging.int_orders
-- unbound preceding is similar to the doing all sum previous values to next value
select salesperson_id,order_number,order_date,cust_id,amount,sum(amount) over (order by order_date rows between unbounded preceding and current row) from staging.int_orders
-- doing lag without lag funtion with help of preceding function
select salesperson_id,order_number,order_date,cust_id,amount,sum(amount) over (order by order_date rows between 1 preceding and 1 preceding ) from staging.int_orders
-- doing lead without lead function with help of following function
select salesperson_id,order_number,order_date,cust_id,amount,sum(amount) over (order by order_date rows between 1 following and 1 following ) from staging.int_orders

WITH d365 as (
SELECT TABLE_SCHEMA,TABLE_NAME,COLUMN_NAME,DATA_TYPE,CHARACTER_MAXIMUM_LENGTH,IS_NULLABLE FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_SCHEMA='STG_EKF_D365' and
TABLE_NAME IN
('EMPLOYEE_V2','TITLES','JOBS','POSITIONS_V2','POSITION_HIERARCHIES','USER_TO_PERSON_RELATIONSHIP',
'VENDOR_CONTACT_INFORMATION','VENDORS_V2','VENDORS_POSTAL_ADDRESSES','VENDOR_SALES_TAX_EXEMPT_NUMBER','VENDOR_BANK_ACCOUNT',
'CUSTOMERS_V3','CUSTOMER_BANKACCOUNTS','CUSTOMER_CONTACT_INFORMATION','CUSTOMER_POSTAL_ADDRESSES_V2','CUSTOMER_SALES_TAX_EXEMPT_NUMBERS',
'APPROVED_VENDOR_LIST_BY_PRODUCTS','EXTERNAL_ITEM_DESCRIPTIONS_FOR_VENDORS_V2','ITEM_BARCODES_V3','PHYSICAL_DIMENSION_DETAILS_BY_PRODUCT',
'SYSCO_DESCRIPTION_TRANSLATIONS','PRODUCTS_V2','PRODUCT_CATEGORY_ASSIGNMENTS','PRODUCT_SPECIFIC_UNIT_CONVERSIONS','RELEASED_PRODUCTS_V2'
)
),
 tgt as (
SELECT TABLE_SCHEMA,TABLE_NAME,COLUMN_NAME,DATA_TYPE,CHARACTER_MAXIMUM_LENGTH,IS_NULLABLE FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_SCHEMA='TGT_EKF_DATA_MIG'
 and TABLE_NAME IN
 ('EMPLOYEE_V2','TITLES','JOBS','POSITIONS_V2','POSITION_HIERARCHIES','USER_TO_PERSON_RELATIONSHIP',
'VENDOR_CONTACT_INFORMATION','VENDORS_V2','VENDORS_POSTAL_ADDRESSES','VENDOR_SALES_TAX_EXEMPT_NUMBER','VENDOR_BANK_ACCOUNT',
'CUSTOMERS_V3','CUSTOMER_BANKACCOUNTS','CUSTOMER_CONTACT_INFORMATION','CUSTOMER_POSTAL_ADDRESSES_V2','CUSTOMER_SALES_TAX_EXEMPT_NUMBERS',
'APPROVED_VENDOR_LIST_BY_PRODUCTS','EXTERNAL_ITEM_DESCRIPTIONS_FOR_VENDORS_V2','ITEM_BARCODES_V3','PHYSICAL_DIMENSION_DETAILS_BY_PRODUCT',
'SYSCO_DESCRIPTION_TRANSLATIONS','PRODUCTS_V2','PRODUCT_CATEGORY_ASSIGNMENTS','PRODUCT_SPECIFIC_UNIT_CONVERSIONS','RELEASED_PRODUCTS_V2'
)
 
 )
select
COALESCE(d.TABLE_SCHEMA,t.TABLE_SCHEMA) as TABLE_SCHEMA,
COALESCE(d.TABLE_NAME,t.TABLE_NAME) as TABLE_NAME,
COALESCE(d.COLUMN_NAME,t.COLUMN_NAME) as COLUMN_NAME,
COALESCE(d.DATA_TYPE,t.DATA_TYPE) as DATA_TYPE,
COALESCE(d.CHARACTER_MAXIMUM_LENGTH,t.CHARACTER_MAXIMUM_LENGTH) as CHARACTER_MAXIMUM_LENGTH,
COALESCE(d.IS_NULLABLE,t.IS_NULLABLE) as IS_NULLABLE
 from d365 d FULL OUTER  JOIN tgt t ON d.TABLE_NAME =t.TABLE_NAME and d.COLUMN_NAME=t.COLUMN_NAME
where t.COLUMN_NAME is NULL or d.COLUMN_NAME is null
